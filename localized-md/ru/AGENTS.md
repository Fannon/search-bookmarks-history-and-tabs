## 1. Сначала подумай, потом кодируй

**Не предполагай. Не скрывай неясности. Выноси компромиссы на обсуждение.**

Перед реализацией:
- Явно сформулируй свои предположения. Если сомневаешься — спроси.
- Если есть несколько вариантов интерпретации, покажи их — не выбирай молча.
- Если есть более простой подход — скажи об этом. Возражай аргументированно, когда это уместно.
- Если что-то непонятно — остановись. Укажи на неясность. Спроси.

## 2. Простота превыше всего

**Минимум кода для решения проблемы. Никакой спекулятивной разработки.**

- Никаких функций сверх того, что было запрошено.
- Никаких абстракций для кода однократного использования.
- Никакой «гибкости» или «настраиваемости», о которой не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Если вы написали 200 строк, а можно было обойтись 50 — перепишите.

Спросите себя: «Сказал бы старший разработчик, что это слишком сложно?» Если да, упростите.

## 3. Прагматичные изменения

**Делайте изменения настолько минимальными, насколько это возможно для достижения наилучшего результата.**

При редактировании существующего кода:
- Предпочитайте точечные изменения, но не будьте настолько хирургически точны, чтобы результат выглядел странно, дублировался или усложнял поддержку.
- Допускается изменять соседний код, тесты или места вызова, если это создает более четкую границу, избавляет от дублирования или упрощает проверку поведения.
- Избегайте попутной очистки, масштабного рефакторинга или смены стиля кода, если это не требуется для выполнения задачи.
- Придерживайтесь существующего стиля кода, даже если вы написали бы иначе.
- Если заметили неиспользуемый код (мертвый код), не связанный с вашими изменениями, сообщите о нем — не удаляйте сами.

Когда ваши изменения оставляют неиспользуемые элементы:
- Удаляйте импорты/переменные/функции, которые стали неиспользуемыми В РЕЗУЛЬТАТЕ ваших изменений.
- Не удаляйте существовавший ранее мертвый код, если об этом не просили.

Критерий: Каждая измененная строка должна иметь вескую причину, связанную с запросом пользователя или сохранением целостности реализации.

## 4. Движение к цели

**Определите критерии успеха. Повторяйте циклы разработки до подтверждения.**

Превращайте задачи в проверяемые цели:
- «Добавить валидацию» → «Написать тесты для некорректных входных данных, затем реализовать код так, чтобы они проходили»
- «Исправить баг» → «Написать тест, воспроизводящий баг, затем исправить код так, чтобы тест проходил»
- «Рефакторинг X» → «Убедиться, что тесты проходят как до, так и после изменений»

Для многоэтапных задач составляйте краткий план:
```
1. [Шаг] → проверить: [метод проверки]
2. [Шаг] → проверить: [метод проверки]
3. [Шаг] → проверить: [метод проверки]
```

## 5. Правила проекта

- Границы: только всплывающее окно браузерного расширения; без фоновых воркеров (background worker), сети или телеметрии; состояние не сохраняется (stateless), за исключением настроек пользователя.
- Стиль: ESM и чистый JS (vanilla JS). Следуйте форматированию Biome: 2 пробела, одинарные кавычки, без точек с запятой.
- Размер: сохраняйте размер расширения минимальным. Не добавляйте зависимости без согласования.
- Поиск: должен оставаться мгновенным на больших объемах данных; предварительно рассчитывайте доступные для поиска поля при загрузке данных.
- Критические пути (Hot paths): избегайте лишней работы с DOM, выделения памяти (allocations), компиляции регулярных выражений на лету и глубокого клонирования объектов.
- Валидация: при изменениях в поиске, расчете оценки, рендере или кэше запускайте соответствующие юнит-тесты и `npm run test:perf`.
- Обработка сбоев: при ошибках API браузера делайте `console.warn` и возвращайте пустые результаты; при сбоях загрузки настроек используйте дефолтные значения и вызывайте `printError`; при сбоях поиска/рендера показывайте скрываемый оверлей ошибок через `printError`.

## 6. Команды

- Запускайте `npm run lint` при изменении кода; используйте `npm run lint:fix` только для автоматически исправляемых проблем в затронутых файлах.
- Предпочитайте точечные тесты перед запуском всего набора:
  - Unit: `npm run test:unit -- path/to/test.js`
  - E2E: `npx playwright test path/to/test.spec.js --project=chromium`
- Запускайте `npm run test:unit` при изменении кода.
- Запускайте `npm run test:e2e` при изменениях в UI или поведении.
- Запускайте `npm run size` при изменении зависимостей, сборки, общих утилит или значительном изменении объема кода.
- Запускайте `npm run build` только при явном запросе или при подготовке релиза.
